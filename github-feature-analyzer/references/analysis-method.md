# 分析方法

## 1. 范围和输入

当任务意图是下面任一情况时使用：

- 解释某个功能的实现逻辑。
- 解释运行机制和边界。
- 比较多个功能的实现原则。
- 在功能深挖前，先从 README 分析项目特征。

必需输入：

- `repo_url`
- `features`，一个或多个功能描述

可选输入：

- `ref`，默认 `main`
- `depth`，`standard` 或 `deep`
- `agent_mode`，`multi`、`single` 或 `auto`

## 2. 获取策略

严格按顺序执行：

1. 先用 MCP 验证仓库访问，并快速探测候选文件。
2. 使用 `scripts/fetch_repo.py` 的 `mcp-first` 模式获取本地源码。
3. archive/API 获取失败时允许回退到 `git clone`。
4. 只有两种获取方式都失败时才停止。

开启历史检索时，优先用 `python3 scripts/reference_retrieval_local.py ...`。它会调用 `scripts/bootstrap_deps.py` 自动创建 `.venv-reference` 并安装依赖；只有用户明确要求 UV 路径时才使用 `scripts/reference_retrieval_uv.sh`。

## 3. 多 agent 拓扑

默认使用多 agent：

1. `overview` agent：仓库概览、入口、模块边界。
2. `architecture` agent：控制面、数据边界、生命周期表面。
3. `feature` agents：每个功能一个，尽量并行。

父 agent 负责：

- 强制子 agent 输出符合 schema。
- 按 `path:line` 去重证据。
- 合并冲突结论并显式标注冲突。
- 把合并产物交给最终报告渲染器。

## 4. 机制优先的分析维度

每个功能最终报告必须覆盖五个维度：

1. 运行时控制流
2. 数据流
3. 状态与生命周期
4. 失败与恢复
5. 并发与时序

函数或文件映射只是证据，不是主体叙述。

## 5. README 优先

功能分析前必须：

- 读取仓库根 README。
- 提取 3 个项目特征或标志性能力。
- 对每个特征给出机制解释和少量证据。
- README 不存在或太弱时，用索引/代码推断，并标记为 `inference`。

## 6. 两阶段功能分析

内部检索/索引阶段：

- 规范化功能关键词。
- 按路径和行级证据给文件打分。
- 过滤弱匹配和噪声。

最终综合阶段：

- 用机制语言直接回答每个功能。
- 用 4-8 句话解释“它是怎么工作的”。
- 按五个维度展开。
- 给每个关键结论附 `path:line` 证据。
- 列出具体风险和未知点。

## 7. 证据规则

每个关键结论都要有证据：

- 首选格式：`relative/path/to/file.ext:line`
- 必要时附短 snippet。

没有直接证据时：

- 标记为 `inference`。
- 说明查找过什么，以及为什么无法确认。

## 8. 降噪规则

- 优先生产运行时代码，少依赖纯文档/注释。
- 只有代码证据不足时才使用文档。
- 不要把内部打分细节放进最终报告。

## 9. 多功能处理

多个功能时：

- 每个功能独立成节。
- 找出共享模块和耦合点。
- 只有证据支持时才报告耦合风险。

当功能涉及 SDK、CLI、子 agent 或工作目录解析时：

- 分类为 `SDK`、`CLI` 或 `Hybrid`。
- 解释工作目录依据，例如 `working_dir`、`current_dir`、`cwd` 或容器映射。
- 没有直接证据时标记为 `inference`。

## 10. 深度模式

`standard`：

- 机制优先回答。
- 五维机制摘要。
- 具体证据和风险。

`deep`：

- 更细的控制流/数据流/生命周期序列解释。
- 有证据的边界检查表。
- 有证据的性能/并发信号。
- 有证据的失败路径和测试信号。

## 11. 质量门槛

合格报告必须满足：

- 包含项目特征部分，并有 3 个项目特征，或明确 fallback。
- 每个功能至少有一个证据项，或明确写“未找到证据”。
- 每个结论映射到具体文件/行。
- 未解决歧义被显式列出。
- 输出遵循 `references/report-schema.md`。
